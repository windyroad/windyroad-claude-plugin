## What Will Surprise You

> Older entries (P103 escalation + above-appetite auto-apply) archived to `afk-subprocess-recovery-archive.md` (2026-05-26 Tier-3 rotation).

- **Iteration-worker subagents can call `ScheduleWakeup` and abandon the synchronous-completion contract** (return "waiting for the wake-up" instead of `ITERATION_SUMMARY`; a `.claude/scheduled_tasks.lock` appears). On a Step 6.75 dirty-for-unknown halt, inspect the uncommitted work first — it may be cosmetic (shippable; advance as recovery) or broken (`git restore` + reopen the slice). (verifying P083)

- **Subprocesses can stall idle 30-90+ min after their last commit; SIGTERM triggers a clean JSON exit-flush** (semantic work done, waiting on a hook timeout or unresolved backgrounded subagent before natural exit). The orchestrator's poll loop should detect "no commits + no JSON output for ≥ N min after last activity" and SIGTERM rather than keep polling. Step 5 backgrounds the subprocess and runs a 60s poll loop with `LAST_ACTIVITY_MARK = max(DISPATCH_START_EPOCH, git log -1 --format=%at HEAD)`; SIGTERM fires when idle exceeds `WORK_PROBLEMS_IDLE_TIMEOUT_S` (default 3600s). Proximate cause (no idle-timeout) fixed; the ultimate cause (whatever the subprocess waits for at exit) remains unsolved and may resurface. (closed P118 / P121 / P086)

- **`/wr-itil:work-problems` Step 5 dispatch is monomodal subprocess by design — presence-detection is unreliable and not the goal; treat the user as transient.** Orchestrator MUST NOT call `AskUserQuestion` between iters except at the framework-prescribed halt points (Step 0 session-continuity / fetch-failure; Step 2.5 / 2.5b loop-end emit; Step 6.5 above-appetite + CI-failure / release:watch halts; Step 6.75 dirty-unknown). Accumulated-question discipline at surface time = direction-setting only / no BUFD / no questions answerable by research (per ADR-044's six-class authority taxonomy). Composes with P122 / P126 / P132 / P135. (verifying P130)

- **P121's SIGTERM-clean-flush guarantee is conditional, not universal.** SIGTERM produces a clean exit-flush only when the subprocess has already emitted `ITERATION_SUMMARY` and is waiting on a final exit hook. If SIGTERM fires while the subprocess hasn't yet emitted ITERATION_SUMMARY (e.g. deadlocked in a `bash until`-loop polling with a regex that never matches — P146; OR **machine-sleep** between polls inflating the **wall-clock** idle-timeout so SIGTERM false-fires on a still-progressing iter — P307), `claude -p --output-format json`'s single-blob-on-exit write has nothing to flush — **treat exit-143 + 0-byte JSON as a metadata-loss event**: verify the work landed via `git log` / `git status --porcelain`, halt the AFK loop, reconstruct cost from the billing dashboard rather than JSON. But if the iter left *coherent uncommitted work* (the P307 machine-sleep case — fix done, tests green, just not committed), **salvage it**: verify via bats, then commit from the orchestrator main turn through a fresh commit-gate review (P261 precedent). SIGTERM still recovers wall-clock; metadata is collateral. (verifying P143; closed P147 / P146; open P307)

- **A Pty-host restart kills in-flight backgrounded iter subprocesses with NO completion notification** (distinct from P147's SIGTERM exit-143 + 0-byte JSON — here there's no exit code at all; the harness lost the detached process). Symptom triad: iter JSON stays **0 bytes**, **no completion notification** arrives, a one-shot `ps` shows **no live `claude -p`**. Recovery: treat as killed-before-commit — verify via `git log`/`git status` (no work lost if the iter never committed; its ticket stays Known Error), discard orphaned edits, re-dispatch. (2026-05-26, P267; sibling of P147.)
