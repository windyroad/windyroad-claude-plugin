# AFK Subprocess Recovery + Mid-Loop Discipline

SIGTERM exit-flush mechanics, iteration-worker discipline (ScheduleWakeup forbidden, Edit/Write blocked workaround), monomodal subprocess dispatch, and mid-loop AskUserQuestion / auto-apply-scorer escalation patterns. Sibling to `afk-subprocess.md` (overview + critical surface) and `afk-subprocess-mechanics.md` (dispatch shape + cost metadata + smaller quirks). Split out 2026-05-03 per P145 MUST_SPLIT contract for Tier 3 budget compliance.

## What Will Surprise You

- **User can escalate from AFK-stop-condition-#2 to interactive AskUserQuestion by asking.** The orchestrator's default in AFK mode is to emit an "Outstanding Design Questions" table (ADR-013 Rule 6 — the persona is away). When the user returns and the session is still alive, they can reply "ask me the questions ... use the askuserquestion tool" and the questions are batched live (cap 4 per call, multiple batches OK). Observed 2026-04-20 AFK loop iter 3 stop: 10 questions answered in 3 batches in ~5 minutes, vs potentially days of asynchronous round-trips if each answer had to prompt a new loop. The `work-problems` final summary could add an affordance line pointing the returning user at this escalation path — see codification candidate 2026-04-20.

- **When release risk is ABOVE appetite and the risk scorer emits a `RISK_REMEDIATIONS:` block, the orchestrator MUST auto-apply scorer remediations incrementally — not escalate to `AskUserQuestion`.** The scorer suggests; the agent decides. Escalating above-appetite decisions the scorer already resolved defeats the AFK purpose — the persona is away and the loop stalls waiting for an answer the scorer already produced. User direction 2026-04-22 iter 4: *"you have a risk scorer to assess release risk - you didn't need to ask me, instead of doing, you wasted time waiting for me to respond"*. Correct flow: extract `RISK_REMEDIATIONS`, read the suggestions, decide what to do, apply it, re-score, drain if now within appetite — else emit Outstanding Design Questions table per ADR-013 Rule 6. See P103. This extends the "act on obvious decisions" memory into the above-appetite release-cadence surface.

- **Iteration-worker subagents (per P077) can call `ScheduleWakeup` and abandon their synchronous-completion contract.** Observed 2026-04-21 AFK iter 5: the worker wrote `packages/itil/skills/transition-problem/SKILL.md` + 2 bats files + a manage-problem forwarder (all sound, 19/19 bats green), then returned "Still running — let me wait for the scheduled wake-up" instead of an `ITERATION_SUMMARY` block. A `.claude/scheduled_tasks.lock` artefact appeared. P077 fixed the delegation mechanism; it did NOT forbid time-deferring primitives in the iteration-worker prompt. **When P036 Step 6.75 fires dirty-for-unknown-reason post-delegation, inspect the uncommitted work first** — run the tests, diff the staged files — the halt may be cosmetic (subagent misbehaved; work is shippable). If sound, advance as halt recovery; if broken, `git restore` + reopen the slice. See P083 for the iteration-worker-must-not-ScheduleWakeup contract gap.

- **Subprocesses can stall idle for 30-90+ minutes after their last commit — and SIGTERM appears to trigger a clean JSON exit-flush.** Observed 2026-04-25 AFK iter 5 (P118): the iteration's 3 commits (`9c50d03`, `257ea5c`, `c8a87fa`) all landed at the ~100-min wall-clock mark; the subprocess then sat for ~70 min in a no-API-work state (RSS oscillating 88-99 MB; no further git commits; 0-byte JSON file) before SIGTERM at the 121-min mark produced a clean 5649-byte response with `is_error: false`, full `## Session Retrospective` section, complete `ITERATION_SUMMARY` block, and `duration_ms: 2992935` (49.9 min — matching only the "real work" portion). The subprocess had completed its semantic work; it was waiting on something (perhaps a hook timeout, perhaps a backgrounded subagent that never resolved) before its natural exit. **Operationally**: the orchestrator's poll loop should detect "no commits + no JSON output for ≥ N min after last activity" and SIGTERM rather than continue polling — the subprocess will produce its output. This evades the retro-on-exit guarantee (P086): retro IS the hang, so it cannot self-surface. Cost from the 2026-04-25 incident: ~$8 of subprocess wall-clock burned for ~$2 of API work after the final commit. **Fixed 2026-04-26 (P121)**: SKILL.md Step 5 now backgrounds the subprocess and runs a 60s poll loop with `LAST_ACTIVITY_MARK = max(DISPATCH_START_EPOCH, git log -1 --format=%at HEAD)`; SIGTERM fires when idle exceeds `WORK_PROBLEMS_IDLE_TIMEOUT_S` (default 3600s = 60 min). Behavioural second-source in `packages/itil/skills/work-problems/test/work-problems-step-5-idle-timeout-sigterm.bats` (fake `claude -p` shim + SIGTERM/JSON-flush/env-override/no-fire assertions). ADR-032 amended 2026-04-26 with the backgrounded-poll-loop refinement. The entry stays here as a "what to watch for if the orchestrator regresses" note — the proximate cause (no idle-timeout) is fixed; the ultimate cause (whatever the subprocess waits for at exit) remains unsolved and may resurface in a different shape.

- **`/wr-itil:work-problems` Step 5 dispatch is monomodal subprocess by design — presence-detection is unreliable and is not the goal; treat the user as transient.** The original P130 framing wanted dual-mode dispatch (subprocess for AFK / main-turn for present); user reframed 2026-04-28: *"you can't tell when I'm AFK or not. I might come in, respond to a question and disappear again ... follow the main purpose of work-problems which is to progress the problem tickets and accumulate questions. When you can no longer make any progress, surface the questions for me to answer."* Fix landed 2026-04-28 (commit `b9da37e`): SKILL.md prose discipline only — new "Mid-loop ask discipline (orchestrator main turn)" subsection inside Non-Interactive Decision Making enumerating the framework-prescribed halt points (Step 0 session-continuity / fetch-failure; Step 2.5 / 2.5b loop-end emit; Step 6.5 above-appetite Rule 5 + CI-failure / release:watch halts; Step 6.75 dirty-for-unknown-reason); orchestrator MUST NOT call `AskUserQuestion` between iters except at those halts; accumulated-question discipline at surface time = direction-setting only / no BUFD / no questions answerable by research-exploration-experimentation (per ADR-044's six-class authority taxonomy); Step 5 iteration-prompt body augmented with transient-user framing. Out-of-scope per the reframe: presence-signal helper, dual-mode dispatch, stream-json observation surface, ADR-032 amendment. ADR-032 subprocess-boundary contract (P084 + P121 + P086 + P089 amendments) preserved verbatim. Behavioural bats `work-problems-no-mid-loop-asking.bats` 20/20 green. The discipline composes with P122 (Step 2.5 default branch), P126 (halt-paths-must-route-design-questions-through-Step-2.5b), P132 (Phase 4 over-ask enforcement), P135 (decision-delegation contract / ADR-044). See P130.

- **P121 SIGTERM-clean-flush guarantee is conditional, not universal.** The subprocess-stall entry above says SIGTERM at 121 min produced a clean 5649-byte JSON exit-flush. Observed 2026-04-29 iter 1 (P143): manual SIGTERM at 68m34s wall-clock produced **0 bytes** — the JSON file stayed empty after exit code 143. The distinguishing factor: in P118's evidence the subprocess had completed semantic work AND emitted ITERATION_SUMMARY before going idle (the JSON was waiting on a final exit hook). In today's case the subprocess deadlocked in a `bash until`-loop polling a backgrounded-task output file with regex `^[0-9]+ tests?,` (bats console-summary format) against a TAP-format file containing `ok 1455 ...` — semantic work was done (commits `e56de11` fix + `e7e471c` retro landed) but ITERATION_SUMMARY had NEVER been emitted, so `claude -p --output-format json`'s single-blob-on-exit write had nothing to flush. **Operational rule**: if SIGTERM-recovery is needed and the subprocess hasn't yet emitted ITERATION_SUMMARY through any visible channel, treat exit-143 as a metadata-loss event — verify the work landed via `git log` / `git status --porcelain`, halt the AFK loop per exit-code semantics, and reconstruct cost from Anthropic billing rather than from JSON. The P121 60-min default + manual mid-loop SIGTERM still recovers wall-clock; metadata is collateral. The polling-regex bug itself is iteration-internal authoring (the agent inside the subprocess wrote a Bash polling shape that didn't match its target output) — separate failure class from P121's stuck-after-emit. **Fixed 2026-05-03 (P147)**: SKILL.md Step 5 now carries the conditional caveat as an explicit "SIGTERM exit-flush is conditional, not universal" subsection that documents the metadata-loss-event handling shape (verify via `git log` + `git status --porcelain`, halt the AFK loop, reconstruct cost from billing dashboard). Behavioural second-source: `packages/itil/skills/work-problems/test/work-problems-step-5-idle-timeout-sigterm.bats` extends with a stuck-before-emit fake-shim asserting `JSON_BYTES=0` after SIGTERM. Mechanism unchanged (SIGTERM remains the recovery primitive); the refinement is documentation accuracy + handling shape. The "polling-regex" iteration-internal authoring bug remains a separate failure class tracked under P146.
