# Releases and CI

Cross-session learnings about push:watch, release:watch, changesets, npm publish, and the GitHub Actions pipeline.

- **npm won't overwrite a published version**. Preview publishes use pre-release suffixes (e.g., `0.1.2-preview.13`) to avoid blocking `changeset publish` from publishing the clean version to `latest`.
- **The `push:watch` script** in root `package.json` is the sanctioned way to push — it runs `git push` then watches the CI run. The git-push-gate hook blocks bare `git push`.
- **Never run `changeset version` locally.** The release pipeline handles versioning. To release: push with `npm run push:watch`, the pipeline creates a release PR, merge it with `npm run release:watch`.
- **`changesets/action@v1`'s `version:` input defaults to running `changeset version` directly**, bypassing any `npm run version` script. To fire a custom version hook (e.g. ADR-021's plugin-manifest sync), explicitly pass `version: npm run version` in `.github/workflows/release.yml`. Observed 2026-04-19 when ADR-021's first production exercise shipped a drifted Version PR because the hook wasn't wired (P052).
- **`release:watch` after `push:watch` MAY require a mid-cycle pipeline rescoring — depends on the bypass kind.** When the commit writes a `reducing-release` bypass marker (auto-applied when the commit closes a problem ticket with tests), the release gate consumes the marker without running drift detection and `release:watch` proceeds. Otherwise the `risk-gate` hook hashes pipeline state at bypass-write; post-commit+push the state hash changes and the hook blocks with `Pipeline state drift`. In that case expect the 3-step dance: `push:watch` → delegate `wr-risk-scorer:pipeline` → `release:watch`. Observed WITHOUT dance 2026-04-20 (4-changeset release via PR #36, all reducing bypasses); WITH dance 2026-04-19 iter 2 (P053) and iter 4 (P048). See P054 for the automation candidate (ADR-independent bypass-kind short-circuit).
- **ADR-022-style contract + migration split as paired commits in one AFK iteration.** When an ADR defers a mechanical migration to a follow-up commit (e.g. ADR-022 Scope lines 71-74), land both commits before pushing so the system is never pushed in an inconsistent state (new SKILL.md contract expecting `.verifying.md` glob + old `.known-error.md` files that match nothing). Iter 5 of 2026-04-19 did this correctly: commit 4e93bcf shipped the contract, immediate follow-up 72eb068 did the 13-file rename, both landed in one `push:watch`.

## What Will Surprise You

- **Release drain fires AT appetite (≥ 4/25), not above it.** The orchestrator Step 6.5 threshold text reads "at or above appetite (4/25)". A score of exactly 4 triggers the drain. Observed 2026-04-20 iter 3: P019 commit scored 4/4/4, drain fired, @windyroad/jtbd@0.6.0 + @windyroad/itil@0.7.2 + @windyroad/architect@0.4.1 released via PR #37. If the score had been 3 no drain would have fired.
- **`push:watch` may report success on the wrong (prior-sha) CI run when an older workflow is still in flight at push time.** The script uses `gh run list --limit 1` which returns the most-recently-STARTED run, not the run for the sha you just pushed. Workaround: after `push:watch` returns, verify `gh run list --commit=$(git rev-parse HEAD) --json name,status,conclusion` shows the HEAD-sha's workflows completed; if not, `gh run watch <ID>` the new-sha runs manually. (The flag is `--commit` / `-c`, not `--head-sha` — `gh run list` has `-c, --commit SHA` but no `--head-sha` flag.) Observed 2026-04-20 AFK iter 4: push:watch reported "Release Preview completed success" but that was the PRIOR sha's workflow; the new sha's CI was still `in_progress`. Same class as P054 (release:watch drift), distinct script. See P060.
- **ADR Confirmation sections that only assert source content (e.g. "`scripts.version` contains X") can pass while the runtime integration silently fails.** ADR-021 asserted its sync script would fire via `npm run version`, but the Release workflow never passed `version:` to the action — first real release produced a drifted PR. When a fix depends on CI behaviour, add behavioural/CI-wiring assertions, not just source greps (P052 revised ADR-021's Confirmation with both).
- **`changesets/action@v1` does NOT tolerate subdirectories under `.changeset/`.** Attempting `.changeset/pending/<held>.md` as a holding area for partial-progress changesets fails the Release workflow with `ENOENT ... /.changeset/pending/changes.md` — the action appears to look for its own internal state files inside the subdirectory and crash on absence. Observed 2026-04-22 after commit e15944f moved the P100 retrospective-minor changeset into `.changeset/pending/`. Fix: hold changesets OUTSIDE `.changeset/` entirely. This repo uses `docs/changesets-holding/` (README banner "Status: provisional" pending P103 / P104 closure); the next Release workflow run after the move passed cleanly. P104 Investigation Tasks track promoting the holding-area to a blessed convention.
- **Local commit accumulation between pushes masks CI regressions on intermediate commits.** GitHub Actions fires CI only on the pushed tip SHA, not on every commit reachable between `@{push}` and `HEAD`. If multiple commits land locally without an intervening push — across interactive sessions, AFK iterations, or cross-session work — a test regression introduced in the middle of the chain is invisible until the next push, at which point CI reports "your commit failed" against the tip SHA even though the tip itself is innocent. Observed 2026-04-24 on P113 commit `b2424c8`: the push batched six commits (`2be1bfa..b2424c8`, including `64f6d3f` from the previous day); tests 699/700/701 in `risk-scorer-structured-remediations.bats` had regressed in `64f6d3f` but origin CI never saw it. My P113 push took the blame; diagnostic `gh run list --branch main` showed runs only for `6b76d13` (last merge) and `b2424c8` — five intermediate commits had no CI record. Follow-up in P116 (push preflight should count `git log @{push}..HEAD` and warn when ≥ 2 commits would batch-push). Adjacent to P040 (origin-fetch preflight) and P109 (work-problems Step 0 prior-session state detection) but distinct — those cover loop-entry posture; this covers cadence across any sequence of local commits.
- **Partial-progress iteration outcomes can paint the release queue into an undrainable corner.** When an iteration lands slice 1 of a multi-slice L/XL ticket and slice 2 is still pending, the resulting changesets may span multiple packages and include a `minor` bump whose stated value proposition only materialises after slice 2. Shipping the minor alone moves adopter layout without the paired feature (Risk "Medium" per risk-scorer Layer 1). Remediations: hold the high-risk changeset in `docs/changesets-holding/` + drain the low-risk subset (this session's R3 path); or defer the whole release until slice 2 ships. Observed 2026-04-22 iter 4 (P100 slice 1 → retrospective minor held; architect + jtbd patches drained; retrospective reinstated + expanded at slice 2). See P103 (orchestrator behaviour) and P104 (structural hazard).
- **Iter retro's "1135/0 bats green" claim does NOT guarantee CI green on the same commit.** Iter retros report local bats results based on whatever the iter ran in its own subprocess; the local invocation may run a subset, may run with different env vars, may benefit from a fixture state CI does not reproduce, or may just be untruthful in the summary prose. Observed 2026-04-26 AFK iter 7 (P065 scaffold-intake): iter reported "Full bats suite 1135 ok / 0 failures" in its ITERATION_SUMMARY notes; CI run `24950750567` on the same pushed commit `8653541` reported `not ok 645 fixture: full re-application is idempotent (no diff)` in `packages/itil/skills/scaffold-intake/test/scaffold-intake-fixture.bats:122`. **Resolved by P127** (commit `482b54a`): the actual root cause was `cp -R . "$TEST_DIR/.snapshot-1"` — GNU coreutils on Linux refuses to copy a directory into itself ("cannot copy a directory, '.', into itself"); BSD `cp` on macOS APFS does NOT detect this case and silently allows it. The non-zero exit from `cp` halted bats on Linux only; macOS reported test green. Fix moved the snapshot directory outside `$TEST_DIR` via a sibling `mktemp -d`. **Operationally**: AFK loops cannot rely on iter retro test claims as release-readiness signal; CI green on the pushed commit IS the signal. The orchestrator's Step 6.5 drain (`push:watch` → `release:watch`) catches CI red on push but cannot recover non-interactively per failure-handling rule. When CI fails on a just-shipped iter commit, the loop halts and the user investigates. Adjacent to P116 (push masks intermediate-commit regressions) but distinct — that's about commits CI never sees; this is about CI seeing differently than local on the same commit.
- **GNU `cp -R . dest` refuses source-into-itself; BSD on macOS allows it silently.** Test authors writing bats fixtures that snapshot a working directory must put the snapshot OUTSIDE the source tree. Pattern: `SNAP=$(mktemp -d); cp -R "$TEST_DIR/." "$SNAP"` rather than `cp -R . "$TEST_DIR/.snap"`. The latter fails on Linux CI (GNU coreutils) but passes on macOS local (BSD), creating exactly the local-vs-CI divergence class P127 captured. Verified 2026-04-26 against both `bats/bats:latest` (Alpine BusyBox `cp` — "recursion detected") and Ubuntu CI (GNU coreutils `cp` — "cannot copy a directory into itself"); both refuse, both with different wording. macOS APFS BSD `cp` allows the operation without warning. Reproduce locally before pushing test fixtures by running `docker run --rm -v "$PWD:/work" -w /work bats/bats:latest <fixture-path>`.
- **Multi-day commit accumulation surfaces latent test-fixture regressions on first push** — and the regression often traces back to an earlier commit in the chain that wiped a fixture source without a matching test sweep. Observed 2026-05-05: a 33-commit batch push surfaced 2 broken bats fixtures on `Run hook tests` CI step (drain-register-queue.bats + bootstrap-catalog.bats); root cause was P168 commit `8edaf7b` ("FFS WIPE THE RXXX risks ... THEY ARE WRONG") wiping `docs/risks/TEMPLATE.md` + renaming R001 without updating dependent test fixtures. Local CI ran fine because each prior session ran the wiped-tree's local-CI subset against the wiped state; remote CI ran against a clean checkout that also lacked TEMPLATE.md but the fixtures cargo-culted `cp $REPO_ROOT/docs/risks/TEMPLATE.md` from the source tree. **Plan**: when wiping a directory or renaming files referenced by tests, sweep `grep -r '<deleted-name>' packages/*/scripts/test packages/*/skills/*/test packages/*/hooks/test` BEFORE the push lands.
