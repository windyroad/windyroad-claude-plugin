# AFK and Subprocess Iteration

Learnings about `/wr-itil:work-problems` AFK loops, `claude -p` subprocess dispatch, and iteration-worker protocols.

> **Sibling briefs** (P145 Tier-3 split): `afk-subprocess-mechanics.md` (dispatch shape, JSON contract, cost metadata, SID quirks) + `afk-subprocess-recovery.md` (SIGTERM flush, iteration-worker discipline, mid-loop escalation). Read all three together.

## What You Need to Know

- **Parallel Claude sessions can create P-ticket numbering collisions.** `manage-problem` / `create-adr` compute `max(local, origin)+1` automatically; `work-problems` fetches origin at loop start. Outside the skills, `git fetch origin` and check the highest P-number before opening tickets.
- **ADR-020 auto-release is skipped inside AFK loops** (ADR-018 Step 6.5, avoids double-release). A fix scoped to non-AFK governance paths is NOT exercised by `/wr-itil:work-problems` iterations — when evaluating Fix Released tickets for closure, check which code path actually fired.
- **Verify Fix Released tickets from your own in-session evidence** before deferring to the user — CI passes, preflight runs, release behaviour, next-ID assignments, commit-gate scoring are all observable. Only leave `needs user verification` for genuinely user-side exercise (interactive invocations not run in session, plugin-cache-lag, other-project-only behaviours). At loop stop, proactively surface close-candidates whose paths were exercised. (verifying P048 + P053)
- **At AFK loop stop, surface outstanding design questions before `ALL_DONE`** — user-answerable blockers (naming, direction, pacing, scope) go in a batched `AskUserQuestion` (cap 4); "needs interactive input" is not terminal. (verifying P053)
- **Every AFK halt path that emits a final summary routes prior-iter accumulated user-answerable skips through Step 2.5b first** (Step 0 continuity/fetch-fail, Step 6.5 CI-fail/above-appetite, Step 6.75 dirty-unknown). Step 2.5b surfaces *prior-iter skips only* — it does NOT ask how to remediate the halt cause itself (those stay halt-with-bug-signal). (verifying P122 + P126)
- **A gate-class hook released mid-AFK-loop is ineffective for the immediate-next iter** — the iter subprocess loads the pre-release cached plugin version (compounds with `claude plugin install` no-op-when-already-installed). Workaround: `/install-updates` between `release:watch` and next-iter dispatch. - **An iter returning `is_error: true` (API stream idle timeout) with coherent staged work is SALVAGEABLE — don't just halt.** Contractual Step 5 branch (ADR-032 § P261 amendment): IF `is_error: true` AND staged files exist AND iter-authored bats pass → the orchestrator commits the staged work from its OWN main turn with iter-attribution (commit gate fires fresh on its own SESSION_ID, per ADR-009); ELSE halt. The salvage commit IS the iteration's one commit. Distinct from P121 SIGTERM (subprocess had committed), P147 stuck-before-emit (0-byte JSON, metadata lost), P146 (bash-polling deadlock). The single bounded exception to "orchestrator does NOT commit from its main turn". (verifying P261) <!-- signal-score: +2 | last-classified: 2026-05-26 | first-written: 2026-05-26 -->

## What Will Surprise You

- **Orchestrator main-turn mid-loop `AskUserQuestion` between iters is forbidden per ADR-044**, even for next-iter target picking — WSJF + ticket-state + tie-break ladder have framework-resolved the selection; asking sub-contracts framework work back. The load-bearing fix is a Phase-2b structural hook (UserPromptSubmit/Stop detecting AskUserQuestion in mechanical-stage zones), beyond the per-skill refactor. (verifying P130 / P132) <!-- signal-score: +2 | last-classified: 2026-05-25 | first-written: 2026-05-17 -->
- **Subprocess dispatch via Bash `run_in_background=true` with `&` kills the subprocess on parent exit** (backgrounded shell HUPs undetached children → 0-byte JSON). Working pattern: put the ENTIRE dispatch+poll loop (`claude -p` + `while kill -0 $ITER_PID` poll + `wait` + cost echo) into ONE shell script run via `run_in_background=true` — the script lives long enough to keep the subprocess attached. <!-- signal-score: +2 | last-classified: 2026-05-25 | first-written: 2026-05-18 -->
- **The runtime-SID create-gate marker (ADR-050) DOES race in the backgrounded work-problems shape** — orchestrator main-turn captures fire PreToolUse hooks concurrently with the iter subprocess; both clobber the per-project runtime-sid marker (last-writer-wins) ⇒ `get_current_session_id` returns the wrong SID ⇒ P119 Write deny. If you hit a P119 deny on a mid-loop ticket Write, it's this known race. Workaround: spam-write `/tmp/manage-problem-grep-<sid>` under recent `/tmp/<system>-announced-*` UUIDs + the runtime-sid value. ADR-050 amended-in-place 2026-05-26 (Option C bounded multi-UUID marker-write = mitigation); impl tracked at P260. <!-- signal-score: +2 | last-classified: 2026-05-26 | first-written: 2026-05-26 -->
- **AFK iter subprocesses can over-claim completion in their `ITERATION_SUMMARY`** — iter 1 of session 8 (P327 / ADR-077 Slice 3) commit message said "all ADR-077 Confirmation items (a)-(j) green at source" yet on-disk the (a)-(j) boxes were all `[ ]` unchecked AND the compendium was NOT regenerated + staged despite Slice 3 implementing exactly that integration in `/wr-architect:review-decisions`. The CI drift gate this iter just shipped failed on the un-regenerated compendium — silent self-contradiction. Cross-check iter claims against on-disk state at Step 6.75 before trusting the summary; consider a `verify-iter-claims` step that greps the iter's stated artefacts against the actual repo. - **Step 0 session-continuity detection has no freshness filter on `.afk-run-state/iter-*.json` is_error markers** — the dir is gitignored and accumulates per-session forever; an iter-4-p246.json with `is_error: true` written 2026-05-18 still fired the Step 0 halt/ask gate on 2026-05-30 (12 days later, P246 verified-closed in the meantime). Workaround: agent reads the marker's mtime + the related ticket's current state and silently proceeds when both signals point to stale-not-load-bearing. Captured as P333.
