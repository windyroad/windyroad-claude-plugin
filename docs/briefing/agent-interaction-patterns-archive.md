## What Will Surprise You (archived)

- **`git mv` + `Edit` + `git add` requires re-stage after the Edit.** `git mv A B` stages the rename; if you then Edit B and `git add <other-files>` before the final commit, only the rename is captured — the Edit to B stays unstaged and lands in the NEXT commit. Must `git add B` after editing. **Enforced by `packages/itil/hooks/p057-staging-trap-detect.sh` (PreToolUse:Bash on `git commit`)**: denies the commit when a staged rename's new path also appears as a working-tree modification, surfacing the `git add <new>` recovery inline. (closed P057; verifying P125)
- **Validate the problem-ticket framing before asking solution-detail design questions.** When a ticket describes a specific solution, the user may have a different shape in mind; the first `AskUserQuestion` should confirm the framing, not dive into details. (Observed twice on P014 — Other-responses kept recovering from framing misses, collapsing scope and eliminating an ADR.) **If a ticket has already been bounced by a framing miss once, open the next AskUserQuestion with a framing check** ("Is this ticket shape still right — pattern, skill, agent, background subagent, or something else?") before detail questions.
- **Problem-ticket Fix Strategy can describe infrastructure that does not exist.** Ticket authors (often run-retro's Step 2b at session-end) describe what they assume exists rather than what the SKILL.md actually contains; the assumption survives ticket creation, WSJF rating, and AFK selection unchallenged. **Always have the architect verify the ticket's premise about existing infrastructure before drafting the fix** — cost of the check is one architect delegation; cost of skipping it is a wasted iteration shipping the wrong shape. Sibling of the framing-validation rule (user-framing vs architect-premise validation). (closed P121)
