# Agent Interaction Patterns

Cross-session learnings about the assistant's own interaction habits — framing-validation, re-stage traps, user-frustration signals, and the small rules that reduce wasted rounds.

> **Sibling brief**: gate-marker propagation, P119 / P122 / P124 helper bugs (P142 fix), and architect-agent gate visibility within AFK iteration subprocesses live in `agent-hook-gate-quirks.md` (split out 2026-05-03 per P145 MUST_SPLIT). Read alongside this file when debugging hook-deny chains.

## What Will Surprise You

- **Write tool fails with "File has not been read yet" after `git mv`.** Renaming a file mid-session invalidates Read-tracking on the new path. `Read` the renamed file before `Write`/`Edit`.
- **User-frustration signal: "why are you asking me and not just doing it?"** When a consent gate or "proceed vs discard?" decision is obvious from direction-pinning, policy (risk within appetite), or homogeneous context, ACT first and report — do not defer. Memory: `feedback_act_on_obvious_decisions.md`. (Observed on the install-updates ADR-030 consent gate when the answer was obviously "all 6 siblings.")
- **When the user has a simple persistence-of-knowledge model in mind, the agent's first instinct may build an over-elaborate framing.** On P167 / RISK-POLICY.md the user's `docs/risks/` model was single-universe ("catalog of per-action risks consumed by per-action assessments — same appetite uniformly"); the agent first built a per-action-vs-lifetime aggregation distinction needing two correction rounds, and the misreading propagated into the architect review and ticket framing. **Propose the SIMPLEST persistence-of-knowledge framing first** ("X is the persistent record consumed by Y") and ask whether they want elaboration before building scope-distinction frameworks. A gate's PASS verdict only proves the elaboration is internally consistent, not that it matches the user's intent. (verifying P167)

- **A multi-question `AskUserQuestion` call can return answers for FEWER than all questions** — observed twice in one session (a 4-question batch returned 3 answers each time; the 4th question's answer was silently absent from the response). The dropped question is not flagged as unanswered — it simply doesn't appear in the returned answers map. When you batch N questions, check the returned answers against the questions you asked and **re-ask any that didn't come back** (a single follow-up `AskUserQuestion` with just the missing one). This is a Claude Code harness behaviour, not a windyroad-plugin defect — no plugin-side fix; the discipline is "verify answer-count == question-count, re-ask the gap." (2026-05-26; ADR-018 dropped from one batch, ADR-043 from another.)

- **On a genuine design fork, lead with your reasoning + a recommended option — don't open with a bare "how do you want to proceed?".** 2026-05-26 RFC pivot: the open form drew *"I want your opinion — what do you think?"*. Present the trade-off + a recommendation (architect-first for architecture forks), THEN `AskUserQuestion` to ratify. Distinct from the act-on-obvious rule above: that's for policy/direction-obvious calls (just act); this is for multi-path forks where the user wants your judgment surfaced first (the ratify ask still fires, just after your recommendation).

- **Decision-confirmation prompts must lead with the substance, not the meta (P302).** When presenting a recorded decision for the user to confirm — `/wr-architect:review-decisions`, `/wr-jtbd:confirm-jobs-and-personas`, and structurally the `/wr-architect:create-adr` Step 5 confirm — open the `AskUserQuestion` `question` field with the one-line Decision Outcome / job statement / persona definition (*what is being decided / asserted*). Sibling-ADR relationships, "separate ADR vs amend" recording-shape, supersession lineage, persona-cluster context: trailing clause or omit. Burying the substance behind meta drove two clarifying re-asks in one session on 2026-05-25 (ADR-045 surfaced as "a sibling ADR codifying five patterns; ADR-038 governs UserPromptSubmit…" → *"if this decision defers to a sibling decision, what does this decision decide??"*; ADR-020 surfaced as "a separate ADR citing ADR-014 + ADR-018 as lineage, keeping single-purpose ADRs" → *"A decision to create a new decision??? What isn't this just the new decision?"*). Both resolved immediately on re-presentation leading with the Decision Outcome. Application of ADR-074's *name the substance, not the grain* principle to the confirm-prompt surface; ADR-026 grounding extended from artifact body to the `AskUserQuestion` `question` text. Rule wording sites: `packages/architect/skills/review-decisions/SKILL.md` Step 3 + `packages/jtbd/skills/confirm-jobs-and-personas/SKILL.md` Step 3.
