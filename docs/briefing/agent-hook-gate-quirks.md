# Agent + Hook Gate Quirks

Cross-session learnings about gate-marker propagation, P119 / P122 / P124 helper bugs, and architect-agent gate visibility within AFK iteration subprocesses. Sibling to `agent-interaction-patterns.md` (which covers the broader interaction-discipline surface). Split out 2026-05-03 per P145 MUST_SPLIT contract for Tier 3 budget compliance.

## What Will Surprise You

- **Bypassing `/wr-itil:manage-problem` by writing directly to `docs/problems/` ships duplicates.** The skill's Step 2 keyword-grep over existing `.open.md` / `.known-error.md` / `.closed.md` filenames + bodies is load-bearing — it catches "this concept is already tracked" before a new file lands. Observed 2026-04-25: the assistant wrote three new `.open.md` files (P119/P120/P121) directly via Write to capture #52831 retrospective findings; all three were near-perfect duplicates of P038 (no voice-tone gate on external comms) and P064 (no risk gate on external comms), both Open since 2026-04-17 with ADR-028 already landed. Cleanup cost: 3 file deletes + 2 evidence-append edits to the surviving tickets. **Always invoke `/wr-itil:manage-problem` even when the observation feels novel** — the grep is fast, the duplicate-bypass cost is high. The retro-skill Step 4b Stage 1 contract makes this mechanical for codification candidates; same discipline applies to ad-hoc ticket creation outside the retro flow.

- **SKILL.md "default for this skill" routing rules can be wrong-by-design when persona ≠ runtime mode.** Observed 2026-04-26 P122: `/wr-itil:work-problems` Step 2.5 routing prose said *"JTBD-006's persona constraint makes the non-interactive path the default for this skill — AskUserQuestion is the exception, not the rule"*. The orchestrator (main turn, with `AskUserQuestion` available, user demonstrably at the keyboard) followed the default and emitted the deferred Outstanding Design Questions table for 12 questions across 5 skipped tickets. User correction within seconds: *"this was a really good example of where there are many outstanding design questions, but nothing was asked"*. The bug: persona (`JTBD-006 = AFK`) was conflated with runtime mode. The runtime detection (subprocess-vs-main-turn per ADR-032; AskUserQuestion-availability; recency-of-prompt) should drive the branch — NOT the skill name. **When a skill prose says "default for this skill is X", verify the default-selection rule is grounded in runtime detection, not in named persona.** When the actual runtime mode disagrees with the named persona, follow the runtime mode. P122 captures the SKILL.md-side fix; the rule generalises to any AFK-named skill that the user invokes interactively.

- **`docs/problems/` create-gate hook (P119) requires the Step 2 marker named with the actual session UUID — `${CLAUDE_SESSION_ID:-default}` evaluates to `default` and the hook denies.** Observed 2026-04-26 P122 retro: the SKILL.md fallback chain produces the wrong marker name; the hook reads `session_id` from its stdin JSON payload (the actual UUID) and the names mismatch. **Resolved by P124 (verifying) 2026-04-26 AFK iter:** `/wr-itil:manage-problem` Step 2 substep 7 now sources `packages/itil/hooks/lib/session-id.sh` and chains to existing `mark_step2_complete()` from `create-gate.sh`. `get_current_session_id()` discovers the canonical UUID via env-var fast path then a fixed system-priority scrape over `/tmp/<system>-announced-<UUID>` markers (architect → jtbd → tdd → itil-assistant-gate → itil-correction-detect → style-guide → voice-tone, write-once per ADR-038). `&&` short-circuits empty-SID writes so an empty-tail marker is never created. Architect-flagged refinement: `-announced-` markers (write-once, no mtime sliding) preferred over `-reviewed-` markers (which `touch`-refresh on every gate check per ADR-009 sliding TTL + P111 subprocess-boundary refresh — picking "most-recent by mtime" can return a stale session's leftover marker). The agent-side ad-hoc scrape pattern is no longer needed in any new manage-problem session. **REGRESSION observed 2026-04-26 P128/P129 capture iter (P124 verification context):** the helper has TWO bugs that defeat its stated guarantee: (1) `shopt -s nullglob` at line 96 errors `shopt: command not found` when `source`'d under zsh — non-fatal but noisy and breaks the intended nullglob-on-no-match contract; (2) the `set -- "${marker_dir}/${system}-announced-"*` glob expands ASCII-alphabetically; without nullglob and with multiple announce markers from different sessions co-resident in `/tmp`, `$1` is the LEXICOGRAPHICALLY-FIRST marker (e.g. `027d3742-...` from Apr 25), NOT the current session's marker (e.g. `1572747b-...` from Apr 26 22:49). Helper returned a stale UUID; subsequent `mark_step2_complete` wrote `/tmp/manage-problem-grep-<stale-UUID>`; the create-gate hook (which reads session_id from its stdin JSON, the REAL UUID) saw no matching marker → denied the Write. Recovery: agent had to manually scrape `ls -lat /tmp/architect-announced-*` for the most-recent timestamp and `touch` the marker under the correct UUID. **The helper's "deterministic priority order" comment claims the within-system ordering is intentional — but it's not session-current, it's ASCII-arbitrary.** Fix shape requires either (a) sort markers by mtime and pick most recent, OR (b) detect and use a current-session-specific signal that doesn't rely on guessing among historical markers, OR (c) use `IFS=$'\n' read -d '' -A` (zsh) / `mapfile -t` (bash) with explicit shell-detection for portability. P124 should bounce back to known-error or sibling ticket should open. Also note: under zsh the glob still expands and the function still returns a UUID — the failure is silent except for the `shopt: command not found` stderr line, so users won't notice the helper is picking the wrong session until the create-gate hook denies their Write. **Resolved 2026-05-03 (P142 Phase 4 / ADR-050 runtime-SID instrumentation hook shipped @windyroad/itil@0.23.3):** new PreToolUse hook fires on every Bash|Write|Edit|Read tool call to write a runtime SID marker file `${TMPDIR}/itil-runtime-sid-${USER}-${PID}.current` containing the actual session_id from the hook stdin JSON; helper reads runtime marker first, falls back to Phase 3 announce-marker priority as cold-path. Eliminates the orchestrator-after-subprocess-dispatch SID-mismatch failure mode. ADR-048 superseded; P144 substep-7 recovery procedure retired. Empirically verified clean in 2026-05-03 post-restart session (3 Write-tool ticket creations succeeded without P144 recovery dance).

- **Architect-agent invocation via the Agent tool from inside an AFK iteration subprocess does NOT always satisfy the architect edit-gate for the SAME subprocess's next Write.** Observed 2026-04-26 P124 AFK iter: ran architect-agent via Agent tool with full design-review prompt; verdict APPROVED with refinements; the next `Write` to `packages/itil/hooks/test/session-id.bats` was blocked by `architect-enforce-edit.sh` with the "Cannot edit ... without architecture review" message. Re-invoked architect-agent (same shape, second call) and the next Write succeeded. The first call's gate-marker presumably wasn't visible to the subprocess's next-tool-use hook chain. May compose with P111 (subprocess tool calls do not refresh parent gate markers — verifying) on the agent-tool axis specifically, or may be a distinct subprocess-vs-parent visibility gap for Agent-tool marker writes (vs the Bash/Tool axis P111 explicitly addressed). **Workaround: when the next Write after an architect-agent call still hits the gate, re-invoke the agent — overhead ~1 turn per affected file.** Distinguish from P107 (closed; that was TTL expiry, not visibility).
- **PostToolUse `architect-mark-reviewed.sh` parses verdict text by literal grep — `"ISSUES FOUND"` substring anywhere in agent output blocks the marker even when the architect's actual verdict on the proposed change is PASS.** Observed 2026-05-10 I002 mitigation declaration: architect's first response opened with heading `**Architecture Review: ISSUES FOUND**` describing structural defects in the *surrounding* ADR landscape (drain-condition empty-conjunct coupling — non-blocking follow-up only); body explicitly concluded `Verdict: PASS on the I002 file write itself. NEW DECISION RECOMMENDED ... do not block the declaration on it.` Hook grep at `packages/wr-architect/<ver>/hooks/architect-mark-reviewed.sh` runs `if echo "$AGENT_OUTPUT" | grep -q "Architecture Review: PASS"; then VERDICT="PASS"; elif echo "$AGENT_OUTPUT" | grep -q "ISSUES FOUND"; then VERDICT="FAIL"`. PASS check ran first but the heading was `ISSUES FOUND`; the elif fired → VERDICT=FAIL → no marker → next Write blocked. Re-emit asked architect (via SendMessage to existing agentId) for `Architecture Review: PASS` heading with NEW DECISION RECOMMENDED note beneath; second response cleared the gate. **Workaround: when the architect's substantive verdict is PASS-with-non-blocking-follow-ups but response opens with `ISSUES FOUND` framing, re-invoke (or SendMessage) asking for `Architecture Review: PASS` heading + follow-up notes underneath.** The hook can't disambiguate "issues in proposed change" vs "issues in surrounding state but proposed change is fine" — the heading IS the signal. <!-- signal-score: 0 | last-classified: 2026-05-11 | first-written: 2026-05-11 -->
