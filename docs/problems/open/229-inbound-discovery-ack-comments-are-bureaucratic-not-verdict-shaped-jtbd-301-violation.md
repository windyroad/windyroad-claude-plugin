# Problem 229: Inbound-discovery ack comments are bureaucratic, not verdict-shaped (JTBD-301 violation)

**Status**: Open
**Reported**: 2026-05-15
**Priority**: 3 (Medium) — Impact: 3 x Likelihood: 1 (deferred — re-rate at next /wr-itil:review-problems)
**Effort**: M (deferred — re-rate at next /wr-itil:review-problems)
**Type**: technical

## Description

The `/wr-itil:review-problems` Step 4.5e safe-and-valid branch posts a gated upstream `gh issue comment` acknowledging the report and linking the local ticket. The current comment shape (observed 31 times across this session's inbound-pipeline processing of upstream issues #125, #126, #124, #123, #121, #120, #117, #110, #98, #87, #86, #85, #84, #97, #83, #82, #81, #80, #79, #78, #77, #76, #63, #62, #61, #60, #59, #58, #57, #56, #42) is:

> Thanks for the report. Tracked locally as `docs/problems/open/<NNN>-<slug>.md` (P<NNN>, classified via `/wr-itil:review-problems` Step 4.5e safe-and-valid branch with safe-low-fix-risk).

User correction screenshot of issue #126's ack comment: *"this is not a suitable response."*

The comment violates JTBD-301's documented desired outcome — *"Submitted reports receive a predictable acknowledgement: labelled, routed, and eventually responded to with a verdict (fix released / parked / duplicate / won't-fix)"*. The current shape:

1. **Leaks internal framework vocabulary to the reporter**: "Step 4.5e safe-and-valid branch with safe-low-fix-risk" is meaningful only to maintainers who know ADR-062. The plugin-user reporter sees jargon, not a verdict.
2. **Carries no verdict**: JTBD-301 names four acceptable verdicts — *fix released / parked / duplicate / won't-fix*. The ack delivers none. "Tracked locally" is processing-paperwork acknowledgement, not a verdict.
3. **Provides no actionable expectation**: the reporter doesn't know when (or whether) the fix will ship, whether to expect updates, whether the ticket is in their downstream tracker's blocked-on list.
4. **Inverts the report-upstream symmetry contract** (ADR-024 / ADR-036): downstream `/wr-itil:report-upstream` produces structured human-language reports; the upstream ack on receipt should mirror that shape (structured + human-language), not maintainer-side framework-vocab.

## Symptoms

- 31 upstream issues received the same boilerplate ack comment with no verdict-shape and no maintainer-vocab translation.
- Visible to any downstream reporter who reads their own filed issue's comments.
- Comment shape was generated by the assistant agent following the prose pattern at `packages/itil/skills/review-problems/SKILL.md` Step 4.5e step 6, which says: *"Then post a gated `gh issue comment` acknowledgement carrying the new local-ticket reference"* — without specifying the verbal shape or the JTBD-301 verdict-shape requirement.

## Workaround

For now: the 31 comments stand. The user-correction surface is this ticket; subsequent fixes happen via the comment-edit mechanism (`gh issue comment <id> --edit` or per-issue follow-up posts). Editing 31 comments is itself friction; the structural fix is to update the SKILL.md template prose so the next inbound-discovery pass produces verdict-shaped comments.

## Impact Assessment

- **Who is affected**: JTBD-301 plugin-user reporter persona. Every adopter who files an inbound report and reads the acknowledgement.
- **Frequency**: every safe-and-valid branch ack (currently the dominant branch). Was 31 this session; ongoing rate scales with inbound report volume.
- **Severity**: High — the JTBD-301 verdict-shape outcome is a primary acceptance criterion for the inbound-discovery contract; violating it across 31 comments at scale undermines the trust contract.

## Root Cause Analysis

### Structural

`packages/itil/skills/review-problems/SKILL.md` Step 4.5e steps 4-6 (above-threshold-pushback / clear-malicious / safe-and-valid branches) name the comment-post action but do NOT prescribe the comment SHAPE. The agent fills in the shape ad-hoc when executing the SKILL. The resulting boilerplate carries internal framework vocabulary because the agent is closer to maintainer-vocab than to reporter-vocab at execution time.

### Why it surfaced now

Inbound-discovery shipped in slices A-G (RFC-004) earlier this session; the inaugural pipeline run (this session) was the first time the comment template was exercised at scale (31 in one batch). The first ack (P198 / #125) was unique-shaped; subsequent acks copied the pattern. The user-correction surfaced after the 31st comment posted.

### Candidate fixes

1. **Add an explicit comment-template block to SKILL.md Step 4.5e step 6**: name the JTBD-301 verdict-shape requirement, prescribe a verdict-shape (e.g., "Acknowledged + tracking as P<NNN> — verdict pending investigation; expect a `fix released` / `parked` / `duplicate` / `won't-fix` update when the local ticket transitions") that the agent fills in per ack. Symmetric for branches 4 (pushback) and 5 (clear-malicious) — each has its own verdict shape (decline / closed-as-malicious).
2. **Voice-and-tone gate run-time enforcement**: the wr-voice-tone agent reviews outbound prose against `docs/VOICE-AND-TONE.md`; the comment template's verdict-shape requirement could be encoded as a voice-tone rule so the gate denies framework-vocab leakage. (Currently the voice-tone gate didn't catch this because the comment passed framework-vocab through — voice-tone reviews tone, not vocabulary-fit per persona.)
3. **JTBD-301 acceptance test**: a behavioural test fixture posts a synthetic ack via the pipeline and asserts the comment body matches the JTBD-301 verdict-shape regex; gates the next pipeline release.

Combination probably right: (1) ships the immediate fix; (3) prevents regression.
